## 5. 処理フロー

### 5.1 CSVインポートフロー

```
[ユーザー] ファイル選択・アップロード
    ↓
[Backend] CSV パース
    ↓
[Backend] バリデーション
    ├─ エラーあり → エラー一覧を返却（ロールバック）
    └─ OK
         ↓
[Backend] プレビューデータを返却
    ↓
[ユーザー] プレビュー確認 → 「インポート実行」
    ↓
[Backend] DB への一括 INSERT（既存データとの重複チェック）
    ↓
[ユーザー] 完了通知
```

### 5.2 D&D割り振りフロー

```
[ユーザー] 左ペインで実績を複数選択
    ↓
[ユーザー] 右ペインの科目箱へドラッグ＆ドロップ
    ↓
[Frontend] PATCH /api/v1/actuals/assign-bulk
           body: { actual_ids: [1, 2], budget_item_id: 42 }
    ↓
[Backend] 対象月の 08_getsukikanryo が 'pending' か確認
    ├─ 'closed' → 422 エラー（締め済みデータは変更不可）
    └─ 'pending'
          ↓
[Backend] 06_shiharaijisseki.budget_item_id / status を更新
    ↓
[Frontend] 科目箱の実績累計・残額をリアルタイム更新
```

### 5.3 月次締めフロー

```
[ユーザー] 月次決算画面で対象月を選択 → 「締め実行」
    ↓
[Backend] 対象月に未割当実績（status='unassigned'）がないか確認
    ├─ あり → 警告表示（強制締め or キャンセル選択）
    └─ なし（or 強制選択）
          ↓
[Backend] 対象月の全実績を 06_shiharaijisseki.status = 'confirmed' に更新
[Backend] 08_getsukikanryo.status = 'closed' に更新
    ↓
[ユーザー] 月次報告書の表示・PDF/Excel出力
```

### 5.4 基金積立フロー（一般会計 → 基金）

```
[ユーザー] 実績割り振り画面で一般会計の歳出実績を「基金積立金」科目に割り振り
    ↓
[Backend] 06_shiharaijisseki.budget_item_id = 基金積立金科目 に更新
    ↓
[ユーザー] 基金管理画面で「積立」を記録
           入力: 対象基金、積立額、日付、メモ
    ↓
[Backend] 09_kikintorihiki レコード作成（type='deposit'）
          deposit_actual_id = 対応する歳出実績のID
```

### 5.5 基金取崩→特別会計歳入フロー（方式B統一）

```
[ユーザー] 基金管理画面で「取崩」を選択
           入力: 対象基金、取崩額、繰入先特別会計、日付、メモ
    ↓
[Backend] 09_kikintorihiki レコード作成（type='withdrawal'）
    ↓
[Backend] 繰入先特別会計に 06_shiharaijisseki レコードを自動作成
           fiscal_year_id = 指定した特別会計の年度ID
           direction = 'revenue'
           raw_category = '基金繰入収入'
           withdrawal_actual_id = 作成した 09_kikintorihiki.id
    ↓
[ユーザー] 特別会計の割り振り画面で
          「歳入 > 基金繰入収入」科目に自動割り当て（確認・修正可）
    ↓
[ユーザー] 特別会計の歳出実績を通常通りCSVインポート → 科目に割り振り
```

### 5.6 暫定予算自動計算フロー

```
[ユーザー] 会計年度設定で「暫定予算作成」を選択
           入力: 暫定期間（月数）
    ↓
[Backend] 05_yosankamoku.is_provisional_target = true の科目を抽出
[Backend] 前年度の各科目の実績合計（06_shiharaijisseki）を集計
[Backend] 暫定予算額 = 前年度実績合計 × (月数 ÷ 12) を計算
    ↓
[Backend] 04_yosanhaibun レコードを一括作成
           budget_type = 'provisional'
    ↓
[ユーザー] プレビュー確認 → 「確定」で保存
```

### 5.7 予算額計算ルール（積算方式 / 案A採用）

事項（行事）の有無によって、目の予算額の計算方法が異なる。

```
目の予算額を求める場合:
    ↓
[判定] 11_gyojiyosanhaibun に当該目(budget_item_id)のレコードが存在するか？
    │
    ├─ 存在する（事項あり目）
    │      → 予算額 = SUM(11_gyojiyosanhaibun.amount
    │                      WHERE budget_item_id = 対象目)
    │        04_yosanhaibun は参照しない
    │
    └─ 存在しない（事項なし目）
           → 予算額 = SUM(04_yosanhaibun.amount
                           WHERE budget_item_id = 対象目)
             ※ 当初予算 + 補正予算の合計
```

**採用理由（案A vs 案B）**:

| | 案A（積算方式）★採用 | 案B（二重管理） |
|---|---|---|
| 目の予算 | `SUM(11_gyojiyosanhaibun.amount)` | `04_yosanhaibun` に別途登録 |
| 整合性 | 自動保証（事項合計 = 目合計） | 手入力のため不一致リスクあり |
| 国の予算書との対応 | 同様（事項合計 = 目合計 = 項合計） | 乖離の可能性 |
| 管理コスト | 低（入力箇所1箇所） | 高（入力箇所2箇所 + チェック） |

**具体例**（旅行関係費 > 交通費 の予算額）:

```
11_gyojiyosanhaibun:
  社会科見学     × 旅行関係費>交通費 = 15,000円
  ホノルルマラソン × 旅行関係費>交通費 = 80,000円
  佐世保旅行     × 旅行関係費>交通費 = 25,000円
                                    ──────────
  → 旅行関係費>交通費 の予算額 = 120,000円（自動計算）
     04_yosanhaibun には旅行関係費>交通費 のレコードなし
```

> **帳票出力時の計算**: 科目別帳票（4.4）・事項別帳票（4.5）のいずれも、
> 目の予算額列は上記ルールで動的に計算する。

### 5.8 立替金の歳出・歳入処理フロー（総計予算主義に基づく）

総計予算主義の原則（歳入歳出は相殺せず全額を計上する）に基づき、立替金は一旦歳出計上し、回収時に歳入計上する。

```
【立替払い時】
[ユーザー] 立替支払いを実施（例: 会社の交際費を立替）
    ↓
[CSV取込] 支払実績（direction='expenditure'）をインポート
    ↓
[ユーザー] 割り振り画面で「立替金 > 会社立替」または「立替金 > 交際費立替」に割り振り
    ↓
[Backend] 06_shiharaijisseki.budget_item_id = 立替金科目 に更新

【精算回収時】
[ユーザー] 立替金の精算回収を受領（口座への振込 or 現金）
    ↓
[CSV取込 or 手入力] 歳入実績（direction='revenue'）をインポート
                    raw_category = '立替返金'
    ↓
[ユーザー] 割り振り画面で「立替返金 > 旅行関係費立替返金」等の
          対応する歳入科目（04_立替返金款）に割り振り
    ↓
[Backend] 06_shiharaijisseki.budget_item_id = 立替返金科目 に更新
```

> **ポイント**: 立替払い額（歳出）と回収額（歳入）は相殺せず、それぞれ全額を計上する。
> 歳出: `立替金 > 会社立替 / 交際費立替`
> 歳入: `立替返金 > 旅行関係費立替返金 / 交際費立替返金 / 会社交通費立替返金`

### 5.9 Discord現金精算フロー

外出先で発生した現金支払いをDiscord Bot経由でADAMS Liteに即時登録する。
BotはDocker Compose内の `discord-bot` コンテナとして動作し、DBには直接アクセスせず、ADAMS LiteのREST API（APIキー認証）を経由して登録する。

**【イベント（スレッド）作成フェーズ】**

```
[ユーザー] Discordで Bot に /setevent イベント名 を送信
    ↓
[discord-bot] スレッドを作成
              「イベント名」スレッドにガイドメッセージを投稿
              例: "🧵 スレッド「2月の買い物」を作成しました。
                   「金額 メモ」の形式で入力してください。"
```

**【現金精算入力フェーズ】**

```
[ユーザー] スレッド内に「金額 メモ」の形式で入力
          例: "500 コンビニ昼食"
    ↓
[discord-bot] メッセージをパース（金額・メモを抽出）
    ├─ 解析失敗 → "入力形式が正しくありません。「金額 メモ」の形式で入力してください" を返信
    └─ 解析成功
          ↓
[discord-bot] 確認テロップをスレッドに返信
              ✅ 以下の内容で登録予定です。
              取引日   ：令和○年○月○日（○）
              金　額   ：500円
              メ　モ   ：コンビニ昼食
              ⏱ 3分後に自動確定します。
              修正する場合は「修正」、すぐに確定する場合は「決定」と返信してください。
    ↓
[discord-bot] タイムアウトタイマー（3分）を開始
```

**【確定フェーズ（タイムアウト / 即時確定 / 修正）】**

```
ユーザーの返信に応じて分岐:

  ─── タイムアウト（3分経過）または「決定」返信 ───
  [discord-bot] POST /api/v1/actuals/discord を呼び出し
               ヘッダー: x-api-key: <APIキー>
               ボディ:
                 date: Discord投稿日時（日本時間）
                 amount: 入力金額
                 note: 入力メモ
                 direction: expenditure（固定）
                 status: unassigned（固定）
                 event_id: スレッド紐付けイベントID（nullの場合もあり）
      ↓
  [backend] 06_shiharaijisseki にレコードを INSERT
            source='discord', status='unassigned'
      ↓
  [discord-bot] "✅ 登録が完了しました。" をスレッドに投稿

  ─── 「修正」返信 ───
  [discord-bot] "何を修正しますか？金額・メモを再入力してください" を返信
  [discord-bot] タイマーをリセットし、新しい入力を待機
               → 新しい入力が来たら【現金精算入力フェーズ】に戻る
```

**【ADAMS Lite側の後続処理】**

```
[ユーザー] 帰宅後、実績割り振り画面（/budget/assign）を開く
    ↓
[Frontend] 未割当実績の一覧を表示
           Discord経由登録は source='discord' で識別可能
    ↓
[ユーザー] 対象科目へドラッグ＆ドロップで割り振り
    ↓
[Backend] 06_shiharaijisseki.budget_item_id / status を更新（5.2 参照）
```

### 5.11 ふるさと納税特別会計フロー

ふるさと納税は暦年（1/1〜12/31）ベースで管理するため、一般会計（4/1〜3/31）とは別に特別会計を設ける。

```
【一般会計での積立】
[ユーザー] 毎月または随時、ふるさと納税に充てる資金を積み立て
    ↓
[ユーザー] 割り振り画面で歳出実績を「ふるさと納税基金繰入 > 繰入金」に割り振り
    ↓
[ユーザー] 基金管理画面で「ふるさと納税基金」に積立（type='deposit'）を記録
    ↓
[Backend] 09_kikintorihiki レコード作成（type='deposit'）
          deposit_actual_id = 対応する歳出実績ID

【ふるさと納税特別会計への繰出・執行】
[ユーザー] ふるさと納税を実施するタイミング（暦年単位）で
          基金から「ふるさと納税特別会計」へ取崩
    ↓
[Backend] 09_kikintorihiki レコード作成（type='withdrawal'）
          → 特別会計の歳入「基金繰入収入」に自動レコード作成（5.5 参照）
    ↓
[ユーザー] ふるさと納税特別会計でふるさと納税の支出実績をCSVインポート
[ユーザー] 割り振り画面で歳出科目（寄附先・品目等）に割り振り
    ↓
[ユーザー] 年次締め（12/31）で特別会計を決算
```

> **資金フロー**:
> ```
> 一般会計（4/1〜3/31）
>   歳出: ふるさと納税基金繰入 > 繰入金
>       ↓ 積立
> ふるさと納税基金（03_kikin）
>       ↓ 取崩
> ふるさと納税特別会計（1/1〜12/31）
>   歳入: 基金繰入収入
>   歳出: ふるさと納税支出（寄附先・品目別に科目設定）
> ```
>
> **なぜ特別会計か**: 一般会計は4月〜翌3月の年度制。ふるさと納税は1月〜12月の暦年制。
> 対象期間が異なるため、一般会計内で管理すると年度またぎが生じる。特別会計に分離することで明確に管理できる。
